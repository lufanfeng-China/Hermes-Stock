# 系统正式规则

## 规则 1：历史数据与当日最新数据的取用规则

### 1.1 目标
当交易系统的判断逻辑同时依赖历史数据与当日最新数据时，必须采用“历史正式数据 + 当日实时临时数据”的融合模式，避免把盘后正式数据与盘中临时数据混为一谈。

### 1.2 数据源优先级
系统中的行情数据源优先级固定为：

1. 本地通达信数据
2. 通达信接口数据
3. 其他公开数据源

适用解释：
- 本地通达信数据：作为历史正式数据主源
- 通达信接口数据：作为盘中实时数据主源
- 其他公开数据源：仅作兜底或补充，不作为主判定依据

### 1.3 历史数据取用规则
对于 T-1 及更早的历史数据，默认全部使用本地通达信数据。

适用范围包括但不限于：
- 日线 OHLCV 历史序列
- 分钟线历史序列
- 均线、MACD、KDJ、RSI 等技术指标底座
- 历史量能均值
- 前高前低、阶段高低点
- 板块历史强弱统计
- 回测、复盘、统计分析

规则要求：
- 历史序列计算默认只认本地通达信正式落盘数据
- 不允许用公开接口历史数据覆盖本地通达信历史数据
- 若本地数据缺失，可使用次级来源补齐，但必须记录来源与补齐标记

### 1.4 当日盘中数据取用规则
若本地通达信数据仅在收盘后更新，则盘中判断时，当日最新数据必须由通达信接口实时补齐。

当日实时补齐可使用的信息包括但不限于：
- 最新价
- 今日开盘价
- 今日最高价
- 今日最低价
- 今日成交量
- 今日成交额
- 分时数据
- 分钟级数据

系统在盘中计算时，应将“历史正式序列”与“当日实时临时 Bar”拼接后再参与计算。

示意：
- 历史正式数据：D1, D2, D3, ..., D(T-1)
- 当日临时数据：D(T)_live
- 计算输入序列：D1, D2, D3, ..., D(T-1), D(T)_live

### 1.5 数据状态定义
系统必须明确区分两类数据状态：

1. final
   - 表示正式收盘数据
   - 来源优先为本地通达信落盘数据
   - 可用于复盘、回测、统计和正式信号确认

2. provisional
   - 表示盘中临时数据
   - 来源优先为通达信接口实时数据
   - 仅用于盘中监控、预警和临时判断

禁止行为：
- 不得将 provisional 数据直接当作 final 数据入库覆盖
- 不得将盘中临时判断与收盘确认结论混用

### 1.6 收盘后替换规则
收盘后，当本地通达信数据完成更新，系统必须执行“临时数据替换正式数据”的切换。

切换规则如下：
- 盘中使用的 D(T)_live 为 provisional
- 收盘后用本地通达信正式落盘的 D(T)_final 替换 D(T)_live
- 当日最终归档、复盘、统计、回测一律以 D(T)_final 为准

这意味着：
- 盘中的结论可以变化
- 收盘后的结论才是正式结论

### 1.7 计算规则
凡涉及“历史 + 当日最新”的指标或交易判断，都必须遵守以下计算原则：

1. 历史部分使用本地正式数据
2. 当日部分使用实时临时数据
3. 输出时必须标记当前判断是否依赖未收盘数据
4. 输出时必须给出数据截止时间
5. 输出时必须给出数据来源

典型场景：
- 今日是否放量突破 20 日线
- 今日 MACD 是否形成盘中金叉
- 昨日涨停后，今日是否高开回封
- 今日量比、换手率、盘口强弱是否异常

### 1.8 盘中信号与收盘确认信号的分离规则
系统必须强制区分两类信号：

#### A. 盘中临时信号（intraday_signal）
适用于：
- 突破均线
- 放量拉升
- 分时回封
- 量比异常
- 分钟级趋势变化
- 盘口强弱变化

规则：
- 可基于实时数据触发
- 必须标记为 provisional
- 仅表示“当前时点成立”
- 不得自动视为收盘确认信号

#### B. 收盘确认信号（close_confirmed_signal）
适用于：
- 日线站稳某均线
- 收盘 MACD 金叉
- 放量阳线正式成立
- 日线形态确认
- 连板结构确认
- 日级别趋势确认

规则：
- 必须基于 final 数据确认
- 收盘前只能给出“可能成立”或“待确认”
- 收盘后才能转为正式结论

### 1.9 输出约束
所有依赖当日未收盘数据的结论，必须附带以下字段或等价信息：
- data_cutoff_time
- data_status
- data_sources
- signal_type
- confirmation_state
- risk_note

推荐语义：
- data_status = provisional / final
- signal_type = intraday_signal / close_confirmed_signal
- confirmation_state = pending / confirmed / invalidated

### 1.10 禁止事项
以下做法在系统中默认禁止：
- 把盘中临时 Bar 当成正式日线写死入历史库
- 用公开数据源历史序列覆盖本地通达信已存在数据
- 在输出中省略“该信号是否收盘确认”的标记
- 用盘中临时值直接驱动复盘结论和回测结果
- 把实时预警信号与收盘选股信号混为一套口径

### 1.11 本规则的正式结论
系统的默认正式规则为：

历史序列计算默认以本地通达信盘后正式数据为准；
当日盘中判断使用通达信接口实时数据生成临时当日 Bar 参与计算；
所有基于当日未收盘数据的结论必须标记为“盘中临时信号”，不得与收盘确认信号混用。

## 规则 2：分钟线、日线、周线的优先级与用途规则

### 2.1 目标
系统必须明确不同时间周期数据各自负责的判断任务，禁止把分钟线、日线、周线混为同一决策层级，导致信号冲突、过度交易或解释混乱。

### 2.2 周期层级定义
系统默认将行情周期划分为三层：

1. 分钟线层
   - 包括 1 分钟、5 分钟、15 分钟、30 分钟、60 分钟等盘中周期
   - 主要用于盘中触发、节奏识别、进出场时机判断

2. 日线层
   - 以单个交易日为基本单位
   - 主要用于主交易判断、趋势判断、结构判断、个股/板块筛选

3. 周线层
   - 以交易周为基本单位
   - 主要用于中期趋势过滤、方向约束、环境定位

### 2.3 决策优先级
系统中的周期优先级固定为：

周线 > 日线 > 分钟线

解释如下：
- 周线决定大方向和中期环境
- 日线决定当前是否具备交易结构与交易理由
- 分钟线只负责触发、择时和执行，不负责推翻更高周期主结论

### 2.4 各周期的职责边界
#### A. 周线职责
周线负责回答：
- 当前标的是中期上升、震荡还是走弱
- 当前是否处于中期高位风险区或低位修复区
- 当前是否允许只做多、轻仓试错、还是应回避

周线适合承载的内容：
- 周级趋势方向
- 周级均线结构
- 周级 MACD / 背离 / 形态背景
- 中期强弱定位
- 中期仓位上限约束

周线不负责：
- 当日是否买入
- 日内具体买点
- 分时级别执行

#### B. 日线职责
日线负责回答：
- 当前这只票/这个板块是否具备可交易结构
- 今天是否处在突破、回踩、转强、衰退等关键阶段
- 是否进入观察池、候选池、回避池

日线适合承载的内容：
- 主趋势判断
- 日线形态判断
- 放量/缩量结构
- 日级别突破、回踩、确认
- 个股筛选与排序
- 板块轮动强弱判断

日线是系统中的主判断周期。
除非另有明确说明，所有“是否值得交易”的主结论默认以日线为核心口径。

#### C. 分钟线职责
分钟线负责回答：
- 什么时候触发
- 从哪里进更合理
- 盘中强弱是否发生变化
- 是否出现应急撤退或止盈信号

分钟线适合承载的内容：
- 盘中突破触发
- 分时回封
- 量比、换手、盘口节奏
- 回踩确认
- 日内买卖点优化
- 风险控制和执行细节

分钟线不负责：
- 单独定义一只票是否值得交易
- 单独推翻周线或日线已经否定的结论

### 2.5 周期冲突时的裁决规则
当不同周期给出冲突信号时，系统必须按以下顺序裁决：

1. 周线否定，日线和分钟线不得单独给出强做多结论
2. 周线中性时，以日线结论为主，分钟线负责优化执行
3. 周线与日线同向时，分钟线信号才可用于积极执行
4. 日线未确认时，分钟线信号只能作为观察或预警，不得直接升级为正式交易信号
5. 分钟线走弱只能影响执行与风控，不能直接改写日线/周线趋势结论

### 2.6 典型使用方式
#### 场景 1：周线看空，日线反弹，分钟线拉升
处理规则：
- 默认视为逆势反弹
- 允许观察，不默认定义为高质量主升交易
- 若要参与，也应降低仓位、缩短持有预期

#### 场景 2：周线中性，日线刚突破，分钟线回踩确认
处理规则：
- 这是较合理的执行场景
- 日线负责给出“可交易”结论
- 分钟线负责给出“何时上车”结论

#### 场景 3：周线向上，日线向上，分钟线盘中转弱
处理规则：
- 大方向不改
- 仅调整执行节奏
- 可以等待更优入场点，或进行短线风控

### 2.7 输出字段约束
凡涉及多周期联合判断，输出中应显式给出：
- weekly_bias
- daily_structure
- intraday_trigger
- timeframe_alignment
- execution_note
- risk_note

推荐语义：
- weekly_bias = bullish / neutral / bearish
- daily_structure = setup_ready / setup_pending / setup_invalid
- intraday_trigger = triggered / waiting / failed
- timeframe_alignment = aligned / mixed / conflicted

### 2.8 信号升级规则
系统中的交易信号升级必须遵守：

1. 周线提供背景过滤
2. 日线提供主结构确认
3. 分钟线提供执行触发

只有当“周线允许 + 日线成立 + 分钟线触发”同时满足时，系统才可将某个机会升级为高优先级执行机会。

如果缺失其中任一层，则应降级处理：
- 缺周线支持：降级为逆势或试错机会
- 缺日线结构：降级为观察中机会
- 缺分钟线触发：降级为待触发机会

### 2.9 禁止事项
以下做法在系统中默认禁止：
- 用分钟线单独生成中期趋势结论
- 用盘中异动直接覆盖日线结构判断
- 用单日强势直接否定周线弱势背景
- 不区分不同周期的职责，混合输出同一条建议
- 只写“看多/看空”，不说明这是周线、日线还是分钟线结论

### 2.10 本规则的正式结论
系统默认采用“周线定方向、日线定结构、分钟线定触发”的三层决策模型。

也就是：
- 周线负责过滤和约束
- 日线负责定义主交易逻辑
- 分钟线负责执行和风控

分钟线不能替代日线，日线不能忽略周线。

## 规则 3：复权、停牌与缺失数据处理规则

### 3.1 目标
系统必须统一处理复权口径、停牌状态和缺失数据问题，避免因为价格口径不一致、停牌误判或脏数据进入计算，导致策略结果失真。

### 3.2 默认价格口径
系统默认采用以下价格口径：

1. 历史结构分析默认使用前复权数据
2. 当日盘中实时判断默认使用未复权实时数据
3. 涉及历史比较时，盘中实时数据必须转换到可比较口径后再参与计算

解释如下：
- 前复权适合趋势、均线、形态、历史高低点等连续性分析
- 未复权实时行情更贴近盘口与成交事实
- 但两者不能直接混算，必须经过统一换算

### 3.3 不同任务的推荐口径
#### A. 趋势与形态类任务
默认口径：前复权

适用范围：
- 均线系统
- MACD / KDJ / RSI
- 历史高低点比较
- 突破、回踩、平台整理
- 波段趋势与结构识别

#### B. 盘口与执行类任务
默认口径：未复权实时价

适用范围：
- 最新价监控
- 分时拉升
- 盘口异动
- 日内买卖点
- 止盈止损触发

#### C. 成交量与换手类任务
默认口径：原始成交量与成交额

适用范围：
- 放量判断
- 缩量判断
- 量比
- 换手率
- 资金活跃度监控

规则要求：
- 成交量和成交额不做复权处理
- 成交类指标必须与价格口径分开管理

### 3.4 口径统一规则
凡涉及“历史序列 + 当日实时数据”的联合计算，必须先统一价格口径，再进行计算。

例如：
- 判断今日是否突破前高
- 判断今日是否站上 MA20
- 判断今日盘中 MACD 是否金叉

统一方式要求：
1. 先明确历史序列使用的是前复权还是不复权
2. 再把当日实时价换算到相同口径
3. 最后才允许拼接进入计算序列

禁止行为：
- 用前复权历史均线直接比较未复权实时价格
- 用未复权实时高点直接比较前复权历史高点
- 不标明价格口径就输出突破/跌破结论

### 3.5 停牌数据处理规则
系统必须显式识别停牌状态，不得把停牌标的当作正常交易标的处理。

停牌状态下的默认规则：
1. 不生成盘中触发信号
2. 不参与盘口强弱比较
3. 不参与实时买点判断
4. 可保留在历史结构分析中，但必须带停牌标记
5. 若停牌天数超过阈值，默认降级或移出短线候选池

建议输出字段：
- trading_status = trading / suspended / resumed
- suspension_days
- last_trading_date
- eligible_for_intraday = true / false

### 3.6 缺失数据处理规则
系统必须区分不同类型的缺失数据，并采用不同策略：

#### A. 关键价格字段缺失
例如：
- open / high / low / close 缺失

处理规则：
- 不允许进入正式技术指标计算
- 不允许输出正式突破/跌破/形态结论
- 可标记为 data_invalid 并进入异常队列

#### B. 成交字段缺失
例如：
- volume / amount / turnover 缺失

处理规则：
- 不允许输出放量、缩量、量比、换手类结论
- 但在价格字段完整时，可保留部分纯价格结构判断

#### C. 辅助字段缺失
例如：
- 行业、题材、财务补充字段缺失

处理规则：
- 不阻断基础行情判断
- 但要降低解释完整度，或在排序时降权处理

### 3.7 缺失补齐规则
当本地主数据缺失时，系统可按优先级尝试补齐：

1. 本地通达信其他可用文件/周期
2. 通达信接口数据
3. 其他公开数据源

补齐后的规则：
- 必须记录补齐来源
- 必须记录补齐时间
- 必须标记该条数据不是主源原始记录
- 涉及关键历史回测时，应允许配置为“禁止补齐，只保留原始主源数据”

推荐字段：
- fill_status = original / backfilled / partial
- fill_source
- quality_score

### 3.8 除权除息事件处理规则
系统必须识别除权除息对价格连续性的影响。

规则要求：
1. 若分析任务依赖历史连续走势，必须优先使用前复权序列
2. 若分析任务依赖真实成交成交价，应保留未复权价格用于执行层
3. 输出中若涉及关键价格位比较，应能说明当前使用的是前复权还是未复权口径

否则会出现以下误判：
- 假突破
- 假跌破
- 均线错位
- 高低点比较失真

### 3.9 输出字段约束
凡涉及复权、停牌、缺失数据处理，输出中应显式给出：
- price_mode
- data_quality
- trading_status
- fill_status
- data_warning
- comparability_status

推荐语义：
- price_mode = qfq / raw / mixed_forbidden
- data_quality = high / medium / low / invalid
- comparability_status = comparable / converted / not_comparable

### 3.10 信号降级规则
当数据口径或数据质量存在问题时，系统必须自动降级信号质量：

1. 价格口径未统一：禁止输出正式信号
2. 标的处于停牌：禁止输出盘中执行信号
3. 关键价格字段缺失：禁止输出结构性结论
4. 成交字段缺失：禁止输出量能类结论
5. 多项数据异常同时存在：直接降级为不可判定

推荐降级等级：
- executable
- observable
- restricted
- invalid

### 3.11 禁止事项
以下做法在系统中默认禁止：
- 将前复权历史序列与未复权实时价格直接拼接计算
- 把停牌标的当作正常活跃标的参与盘中比较
- 对缺失关键字段的数据继续输出正式交易建议
- 不标注价格口径就输出前高、前低、突破、均线站上等结论
- 在回测与实时系统中混用不同复权口径且不说明

### 3.12 本规则的正式结论
系统默认采用“历史分析以前复权为主，盘中执行以未复权实时价为主，所有跨时点比较必须先统一口径”的处理方式。

同时：
- 停牌标的不参与盘中执行判断
- 缺失关键字段的数据不进入正式信号计算
- 所有补齐数据都必须保留来源与质量标记

## 规则 4：个股、板块、指数三层信号的约束关系

### 4.1 目标
系统必须明确指数环境、板块强弱、个股信号三者之间的层级关系，避免只看单票异动、忽略板块与市场背景，导致交易结论失真。

### 4.2 三层对象定义
系统默认把交易判断拆成三层：

1. 指数层
   - 代表大盘环境、风险偏好、市场宽度、赚钱效应
   - 回答“现在整体能不能做、适合做什么风格”

2. 板块层
   - 代表行业、题材、概念、热点主线与轮动方向
   - 回答“当前应该优先做哪个方向”

3. 个股层
   - 代表具体交易标的
   - 回答“这只票是否值得观察、跟踪、执行”

### 4.3 决策优先级
系统中的三层优先级固定为：

指数层 > 板块层 > 个股层

解释如下：
- 指数层决定整体风险预算和风格允许范围
- 板块层决定方向选择和资金聚焦点
- 个股层决定具体执行对象

个股信号不能脱离指数与板块背景独立解释。

### 4.4 各层职责边界
#### A. 指数层职责
指数层负责回答：
- 当前市场是进攻、轮动、防守还是回避
- 当前市场宽度是否支持积极出手
- 当前风格偏大票、小票、成长、价值还是题材炒作
- 当前仓位上限应偏高、偏中还是偏低

指数层适合承载的内容：
- 市场温度
- 市场宽度
- 指数趋势结构
- 涨跌停家数、上涨下跌比例
- 风险偏好与风格偏好
- 总体仓位约束

指数层不负责：
- 指定某只票的精确买点
- 直接决定哪只个股一定可做

#### B. 板块层职责
板块层负责回答：
- 当前最强方向是什么
- 当前方向处于启动、发酵、高潮还是退潮
- 哪些板块具备持续性，哪些只是脉冲
- 个股属于主线、支线还是杂毛跟风

板块层适合承载的内容：
- 板块热度
- 板块生命周期
- 龙头与跟风结构
- 板块内部一致性
- 轮动强弱和持续性

板块层不负责：
- 在没有个股结构的前提下直接给出执行动作
- 用单个龙头的走势替代整个板块结论

#### C. 个股层职责
个股层负责回答：
- 这只票是否符合当前市场与板块背景
- 这只票在板块中的角色是什么
- 这只票的日线结构和分钟线触发是否具备执行条件
- 这只票当前应归入观察、候选还是回避

个股层适合承载的内容：
- 个股结构
- 个股角色定位
- 个股触发条件
- 个股失效条件
- 个股仓位建议
- 个股执行计划

### 4.5 三层约束规则
系统必须按以下顺序应用约束：

1. 先判断指数层是否允许出手
2. 再判断板块层是否存在明确主线或可做方向
3. 最后判断个股层是否具备可执行结构

只有三层依次通过时，个股机会才可升级为高优先级机会。

### 4.6 指数层对板块层的约束
当指数层给出不同环境时，板块层默认受以下约束：

#### A. 指数层偏进攻
处理规则：
- 允许优先关注高弹性主线方向
- 允许追踪强势题材与高辨识度板块
- 板块评分上可提高趋势持续性权重

#### B. 指数层偏轮动
处理规则：
- 板块选择应偏向强度稳定、节奏清晰的方向
- 不鼓励在高潮末端盲目追高
- 更重视分歧转一致、回踩确认类机会

#### C. 指数层偏防守或回避
处理规则：
- 板块层即使出现局部热点，也不得默认提升为高仓位主攻方向
- 板块机会应整体降级
- 更强调防守、等待或轻仓试错

### 4.7 板块层对个股层的约束
个股层必须服从板块层，不得只因单票异动就脱离板块背景独立升级。

默认规则：
1. 板块处于启动/发酵期，个股强结构可积极跟踪
2. 板块处于高潮期，个股信号必须提高风险折扣
3. 板块处于退潮期，个股即使短时走强，也默认降级处理
4. 脱离主线板块的个股，除非有非常明确的独立逻辑，否则不升级为核心候选
5. 个股在板块中的角色不同，优先级必须不同

个股角色建议划分：
- leader：龙头/总龙
- core：核心中军
- pivot：辨识度强的趋势核心
- follower：普通跟风
- noise：杂毛/弱相关

### 4.8 个股升级规则
个股要被升级为“高优先级执行机会”，默认必须同时满足：
- 指数层不否定
- 板块层有明确方向且未明显退潮
- 个股处于板块内较高角色层级
- 个股自身日线结构成立
- 个股分钟线触发成立

若缺少其中任一项，则降级处理：
- 缺指数支持：降为受限机会
- 缺板块支持：降为孤立机会
- 缺角色优势：降为普通机会
- 缺结构成立：降为观察机会
- 缺触发确认：降为待触发机会

### 4.9 孤立强势股处理规则
系统允许识别“孤立强势股”，但默认不把它与主线强势股同等对待。

默认判断：
- 若指数弱、板块弱、只有个股强，则定义为孤立强势
- 孤立强势可进入观察名单，但默认不进入核心主攻池
- 若后续形成板块扩散或市场风格支持，才允许升级

### 4.10 输出字段约束
凡涉及指数、板块、个股三层联动判断，输出中应显式给出：
- market_regime
- market_risk_budget
- sector_strength
- sector_lifecycle
- stock_role
- stock_setup_status
- layer_alignment
- action_grade

推荐语义：
- market_regime = aggressive / rotation / defensive / no_trade
- sector_lifecycle = ignition / expansion / climax / decay
- stock_role = leader / core / pivot / follower / noise
- layer_alignment = aligned / partially_aligned / conflicted
- action_grade = core / standard / exploratory / avoid

### 4.11 信号降级规则
当三层出现不一致时，系统必须自动降级：

1. 指数否定：个股最高只能降级为观察或试错
2. 板块退潮：个股最高只能降级为低优先级机会
3. 个股只是跟风：不得升级为核心机会
4. 三层冲突明显：直接进入回避或等待状态

### 4.12 禁止事项
以下做法在系统中默认禁止：
- 只看个股分时拉升，不看指数和板块背景
- 仅凭单只龙头上涨就认定整个板块可做
- 在指数极弱环境下仍按常规高仓位对待个股强信号
- 不区分龙头、核心、跟风、杂毛而给出相同级别建议
- 输出个股建议时不说明其市场环境与板块归属

### 4.13 本规则的正式结论
系统默认采用“指数定环境、板块定方向、个股定执行”的三层约束模型。

也就是：
- 先看市场能不能做
- 再看做什么方向
- 最后看具体做哪只票

如果指数、板块、个股三层不一致，个股信号必须降级，不得直接升级为核心执行机会。

## 规则 5：仓位控制、信号分级与执行动作规则

### 5.1 目标
系统必须把“看法”与“动作”分开管理，不允许仅凭方向判断直接生成同等强度的执行建议。所有交易结论都必须映射到明确的信号等级、仓位上限与执行动作。

### 5.2 核心原则
系统默认遵循以下原则：

1. 先有环境判断，再有机会判断，再有执行动作
2. 仓位大小取决于信号质量，而不是主观信心
3. 同一信号在不同市场环境下，对应的仓位上限不同
4. 未确认信号不能使用确认后仓位
5. 风险控制优先级高于进攻性加仓

### 5.3 动作分层
系统默认把执行动作分为五层：

1. avoid
   - 回避，不开仓
2. observe
   - 观察，不执行，仅跟踪
3. probe
   - 试错，小仓位尝试
4. standard
   - 标准执行，正常仓位
5. core
   - 核心执行，允许较高优先级配置

动作层级不等于看多强度，而是“允许采取多大力度的操作”。

### 5.4 信号等级定义
系统默认把机会质量分为四级：

1. S 级
   - 指数、板块、个股三层高度一致
   - 周线/日线/分钟线协同
   - 结构清晰，角色领先，触发明确

2. A 级
   - 主要条件成立，只有少量瑕疵
   - 可执行，但需要常规风险折扣

3. B 级
   - 条件部分成立
   - 更适合观察或轻仓试错

4. C 级
   - 条件冲突明显或证据不足
   - 默认不执行，最多保留观察

### 5.5 信号等级与默认动作映射
默认映射关系如下：

- S 级 -> core / standard
- A 级 -> standard
- B 级 -> probe / observe
- C 级 -> observe / avoid

但最终动作仍需受市场环境、板块状态、数据质量、盘中/盘后确认状态共同约束。

### 5.6 仓位上限规则
系统必须给出“信号级仓位上限”和“组合级仓位上限”两层约束。

#### A. 单笔/单标的仓位上限
建议默认分层：
- avoid：0%
- observe：0%
- probe：不高于计划标准仓位的 25%
- standard：不高于计划标准仓位的 100%
- core：不高于计划标准仓位的 100%，但允许在组合约束内优先分配

#### B. 组合级仓位上限
组合上限必须先受指数环境约束，再受机会数量与集中度约束。

默认原则：
- 市场偏进攻：组合上限可较高
- 市场偏轮动：组合上限中性，强调分散与节奏
- 市场偏防守：组合上限较低
- no_trade：默认空仓或极低试错仓

### 5.7 仓位决定顺序
每次生成执行建议时，系统必须按以下顺序决定仓位：

1. 指数层决定组合总风险预算
2. 板块层决定方向分配比例
3. 个股层决定单票优先级
4. 信号等级决定该票理论仓位上限
5. 盘中/盘后确认状态决定是否打折
6. 数据质量与风险项决定是否进一步降级

### 5.8 加仓规则
系统默认不允许“因为盈利或兴奋而主观加仓”，只能按规则加仓。

允许加仓的前提：
- 初始仓位已按规则建立
- 后续出现新增确认信号
- 指数环境未恶化
- 板块未退潮
- 个股未出现失效信号
- 加仓后组合总风险仍在上限内

推荐加仓类型：
1. 突破后回踩确认加仓
2. 分歧转一致加仓
3. 板块强度继续扩散加仓
4. 收盘确认后由盘中试错仓升级为标准仓

### 5.9 减仓与退出规则
系统必须把减仓和退出视为与开仓同等重要的正式动作。

默认减仓触发：
- 市场环境转弱
- 板块生命周期由强转弱
- 个股触发失败
- 个股跌破关键结构位
- 成交与强度明显不及预期
- 达到预设目标区但未继续强化

默认退出触发：
- 失效条件成立
- 数据层发现关键风险或异常
- 多层信号由一致转为明显冲突
- 风险收益比恶化到不可接受

### 5.10 盘中信号与仓位折扣规则
盘中临时信号不得直接对应满仓或标准满配动作。

默认折扣原则：
- provisional 信号最多对应 probe 或折扣后的 standard
- close_confirmed_signal 才可升级到完整标准动作
- 若只是分钟线触发而日线未确认，不得给 core 动作

### 5.11 风险项对动作的否决权
以下风险项具有优先否决权：
- 数据不可比或质量低
- 停牌/异常状态
- 指数层明显不支持
- 板块退潮明显
- 个股角色过弱
- 流动性不足
- 失效位过近导致赔率不足

一旦命中高优先级风险项，系统必须把动作至少降一级，必要时直接转为 avoid。

### 5.12 输出字段约束
凡涉及执行建议，输出中应显式给出：
- signal_grade
- action_grade
- position_cap_single
- position_cap_portfolio
- add_on_condition
- reduce_condition
- invalidation
- execution_state

推荐语义：
- signal_grade = S / A / B / C
- action_grade = avoid / observe / probe / standard / core
- execution_state = pending / active / reduced / exited

### 5.13 禁止事项
以下做法在系统中默认禁止：
- 只给“看多/看空”而不给动作等级
- 在盘中未确认状态下直接给出满配执行建议
- 忽略指数环境直接按单票强弱决定仓位
- 板块退潮时仍维持主升仓位口径
- 把观察信号和执行信号使用同样仓位
- 不写失效条件就给出加仓建议

### 5.14 本规则的正式结论
系统默认采用“信号分级 -> 动作分级 -> 仓位映射 -> 风险折扣”的执行框架。

也就是：
- 先判断机会质量
- 再决定允许采取什么动作
- 再决定最多能给多少仓位
- 最后由确认状态和风险项继续折扣

任何没有明确等级、仓位上限、失效条件的信号，都不能直接进入正式执行层。

## 规则 6：规则引擎输出格式与 AI 输出边界

### 6.1 目标
系统必须严格区分“规则引擎负责产出的结构化结论”和“AI 负责产出的解释性内容”，避免 AI 直接替代规则判断，或把不可追溯的自然语言混进正式交易结论。

### 6.2 总体原则
系统默认遵循以下原则：

1. 规则引擎先产出，AI 后解释
2. 所有可执行结论必须先有结构化规则结果
3. AI 不得生成脱离规则结果的正式交易动作
4. AI 可以解释、总结、重写、提示风险，但不能私自改写规则核心字段
5. 输出给用户时，必须能一眼区分“规则结论”和“AI 解读”

### 6.3 双层输出模型
系统默认采用双层输出结构：

1. rule_based_output
   - 由规则引擎生成
   - 结构化、可追溯、可比较、可回测

2. ai_summary
   - 由 AI 生成
   - 面向阅读、解释、归纳、提醒
   - 不作为唯一执行依据

### 6.4 规则引擎必须负责的内容
以下内容必须由规则引擎直接产出，不得只由 AI 自由生成：
- market_regime
- market_risk_budget
- sector_lifecycle
- stock_role
- signal_grade
- action_grade
- position_cap_single
- position_cap_portfolio
- entry_trigger
- add_on_condition
- reduce_condition
- invalidation
- signal_type
- confirmation_state
- price_mode
- trading_status
- data_quality
- data_cutoff_time
- data_sources

规则要求：
- 以上字段必须结构化
- 必须可被程序消费
- 必须可用于排序、筛选、回测和审计

### 6.5 AI 允许负责的内容
AI 默认允许负责以下内容：
- 把结构化字段翻译成自然语言
- 总结为什么当前是这个结论
- 用更易读的方式解释风险、条件和失效点
- 对多个规则信号做摘要归纳
- 给出阅读提示和注意事项
- 生成日报、复盘说明、简报风格摘要

AI 的任务重点是：
- 提高可读性
- 提高理解效率
- 提高信息压缩能力

而不是替代规则判断。

### 6.6 AI 默认禁止的内容
以下内容在系统中默认禁止由 AI 单独决定：
- 直接改写 signal_grade
- 直接改写 action_grade
- 在没有规则字段支撑时新增买点/卖点
- 在没有规则确认时把 observe 升级成 standard 或 core
- 覆盖 invalidation
- 覆盖 position_cap_single / position_cap_portfolio
- 省略关键风险项后输出偏乐观结论
- 用模糊措辞掩盖规则冲突或数据问题

### 6.7 最终输出结构要求
面向前端或 API 的最终输出，必须同时包含两部分：

#### A. 结构化规则结果
至少包含：
- rule_based_advice
- key_fields
- signal_grade
- action_grade
- invalidation
- data_meta

其中：
- rule_based_advice 用于简洁表达规则结论
- key_fields 用于承载完整结构化字段集合
- data_meta 用于承载时间戳、来源、状态、价格口径等元数据

#### B. AI 解释结果
至少包含：
- ai_summary
- ai_risk_note
- ai_focus_points

要求：
- AI 内容必须与规则结果并列展示
- 不能覆盖规则字段
- 不能作为唯一输出层

### 6.8 字段冲突处理规则
如果 AI 文本与规则字段产生冲突，系统必须以规则字段为准。

默认处理方式：
1. 保留规则字段原值
2. 标记 AI 输出冲突
3. 必要时触发 AI 重生成
4. 若冲突仍存在，则前端优先展示规则结果，并弱化或隐藏冲突 AI 文本

推荐字段：
- ai_consistency_status = consistent / conflicted / regenerated
- ai_override_allowed = false

### 6.9 输出模板规则
系统默认推荐如下展示顺序：

1. 规则结论摘要
2. 信号等级与动作等级
3. 条件 / 风险 / 失效
4. 数据时间、来源、状态
5. AI 摘要解释
6. AI 风险提示

这样可以确保：
- 用户先看到可执行口径
- 再看到解释性文字
- 不会把 AI 的润色文本误当成正式规则

### 6.10 审计与回测规则
所有进入正式信号流的结果，都必须以规则引擎字段为审计主依据。

规则要求：
- 回测只使用结构化规则字段，不使用 AI 文本
- 审计只以规则字段和原始数据为准
- AI 内容只作为辅助解释材料保存
- 若 AI 文本缺失，不影响系统正式判断与回测

### 6.11 页面级输出要求
#### A. 页面一：市场总览
必须先给：
- market_regime
- market_risk_budget
- breadth / style / risk metrics

AI 只负责解释“为什么今天是这个市场环境”。

#### B. 页面二：热点题材
必须先给：
- topic_heat_score
- topic_lifecycle
- leaders
- watchlist
- candidate_pool
- avoid_list

AI 只负责解释“当前主线逻辑、题材节奏和风险点”。

#### C. 页面三：个股/策略候选
必须先给：
- valuation_score 或 strategy_score
- signal_grade
- action_grade
- entry / add / reduce / invalidation

AI 只负责解释“为什么这只票当前被放在这个层级”。

#### D. 页面四：个股诊断/报告
必须先给：
- 结构化诊断字段
- 风险结论
- 失效条件
- 执行等级

AI 只负责把这些内容整理成可读报告。

### 6.12 禁止事项
以下做法在系统中默认禁止：
- 只有 AI 摘要，没有规则字段
- 让 AI 直接生成买卖动作并绕过规则引擎
- 把 AI 文本回写成正式结构化字段
- 在前端只展示 AI 结论，不展示规则依据
- 回测时使用 AI 文本作为判断依据
- 用户看到“建议买入”，却看不到 signal_grade、action_grade、失效条件和数据状态

### 6.13 本规则的正式结论
系统默认采用“规则引擎产出正式结论，AI 产出解释性摘要”的双层输出架构。

也就是：
- 规则负责判断
- AI 负责解释
- AI 不能替代规则
- AI 与规则冲突时，以规则为准

任何没有结构化规则字段支撑的 AI 结论，都不能直接进入正式交易决策层。

## 规则 7：数据刷新频率、缓存失效与计算触发规则

### 7.1 目标
系统必须明确不同数据层与规则层的刷新节奏、缓存策略和重算触发条件，避免实时数据过度重算、日级数据重复计算、前后端口径不一致，或因为缓存滞后导致错误判断。

### 7.2 总体原则
系统默认遵循以下原则：

1. 刷新频率必须与数据时效性匹配
2. 计算频率不能高于其输入数据的有效更新频率
3. 能缓存的结果尽量缓存，但缓存必须可解释、可失效、可追踪
4. 盘中结果允许短缓存，盘后正式结果必须稳定固化
5. 任何缓存结果都必须携带生成时间与底层数据时间

### 7.3 四层刷新模型
系统默认采用四层刷新频率：

1. 实时层（约 10~30 秒）
   - 用于指数快照、盘口快照、盘中触发状态、热点快照

2. 短周期层（约 3~5 分钟）
   - 用于板块热度、观察池、候选池、短周期评分、环境标签

3. 日级层（收盘后 / 每日一次主刷新）
   - 用于日线指标、复盘结果、正式规则结论、候选池固化、诊断归档

4. 周级层（每周一次主刷新）
   - 用于阈值校准、行业基线、统计回顾、策略参数校验

### 7.4 各类数据的默认刷新频率
#### A. 指数与市场快照
默认刷新：10~30 秒

适用内容：
- 指数涨跌幅
- 市场宽度快照
- 涨跌停家数快照
- 风险偏好快照
- 盘中环境标签

#### B. 板块与题材热度
默认刷新：3~5 分钟

适用内容：
- 板块热度分数
- 生命周期状态
- 龙头/跟风结构
- watchlist / candidate_pool 初步更新

#### C. 个股执行信号
默认刷新：30 秒到 5 分钟之间，取决于信号类型

细分建议：
- 分钟线触发信号：30~60 秒
- 日线结构状态：3~5 分钟或关键事件触发
- 执行动作建议：随触发结果联动更新

#### D. 日级正式结果
默认刷新：收盘后一次主刷新

适用内容：
- final 日线数据
- 收盘确认信号
- 正式候选池
- 正式个股诊断
- 当日归档、复盘、统计

### 7.5 缓存分层规则
系统默认把缓存分成三层：

1. 原始数据缓存
   - 缓存源数据请求结果
   - 适用于指数、行情、板块列表、个股快照

2. 中间计算缓存
   - 缓存已标准化或已预处理的结果
   - 适用于统一字段后的市场快照、板块热度表、个股评分表

3. 最终结果缓存
   - 缓存规则引擎最终输出
   - 适用于页面接口返回、候选池列表、个股诊断报告

默认原则：
- 原始数据缓存时间最短
- 中间计算缓存次之
- 日级 final 结果缓存可以最长，但必须在新交易日或新正式数据到达时失效

### 7.6 缓存元数据要求
所有缓存对象都必须至少携带以下元数据：
- generated_at
- source_data_cutoff_time
- source_data_status
- cache_ttl
- cache_scope
- cache_key_version

推荐语义：
- source_data_status = provisional / final
- cache_scope = raw / intermediate / final

### 7.7 主动失效规则
以下事件必须触发缓存主动失效或重算：

1. 新交易日开始
2. 本地通达信盘后正式数据更新完成
3. 关键上游数据源返回更近时间戳数据
4. 板块生命周期状态切换
5. 个股信号状态由 pending -> triggered / invalidated
6. 用户切换价格口径、周期口径或筛选条件
7. 规则版本变更

### 7.8 被动失效规则
即使没有显式事件，缓存也必须按 TTL 自动失效。

默认建议：
- 实时层缓存 TTL：10~30 秒
- 短周期层缓存 TTL：3~5 分钟
- 日级正式结果 TTL：到下一交易日开盘前或直到新 final 数据到达
- 周级统计结果 TTL：到下一次周级刷新前

### 7.9 计算触发规则
系统中的计算不应固定全量轮询，必须支持事件触发 + 定时触发结合。

默认触发方式：
1. 定时触发
   - 适用于市场快照、板块热度、观察池周期刷新

2. 数据到达触发
   - 适用于本地通达信盘后数据更新后，重算 final 日级结果

3. 状态变化触发
   - 适用于个股由未触发变为触发、板块由发酵转高潮、市场由轮动转防守

4. 用户查询触发
   - 适用于低频个股诊断、单票深度分析、临时筛选任务

### 7.10 重算范围控制规则
重算必须分级，禁止默认全系统全量重算。

默认规则：
- 指数变化：优先重算指数层与受影响的环境标签
- 板块变化：优先重算对应板块及其成分候选
- 个股触发变化：优先重算该个股及相关列表排名
- 本地正式数据更新：重算全部受日级数据影响的正式结果
- 周级参数更新：重算阈值与统计，不必立即全量重算全部盘中缓存

### 7.11 盘中与盘后切换规则
系统必须显式区分盘中运行模式与盘后运行模式。

#### A. 盘中模式
默认特点：
- 允许 provisional 数据
- 允许短 TTL
- 允许频繁更新
- 输出以监控、触发、跟踪为主

#### B. 盘后模式
默认特点：
- 只认 final 数据
- 进行正式归档
- 固化当日候选、评分、诊断
- 供复盘、回测、日报使用

盘后模式一旦完成，当日正式结果不得再被盘中 provisional 缓存覆盖。

### 7.12 前后端一致性规则
前端轮询频率不能脱离后端缓存与重算节奏。

默认要求：
- 前端轮询频率不得显著快于后端更新频率
- 若后端 30 秒才更新一次，前端不应每秒请求
- 接口响应中应返回缓存状态与数据时间
- 前端必须区分“新结果未产生”和“接口失败”两种状态

推荐字段：
- refresh_interval_hint
- data_cutoff_time
- cache_status
- next_refresh_after

### 7.13 历史回测与正式归档规则
回测、复盘、日报和归档只能使用日级或周级正式结果，不得读取盘中短缓存结果作为正式历史依据。

规则要求：
- 回测只读 final 数据与正式规则输出
- 盘中缓存结果仅用于实时监控，不进入正式历史样本
- 若需要保留盘中过程，只能作为事件日志单独保存，不得与正式结论混表

### 7.14 禁止事项
以下做法在系统中默认禁止：
- 用同一缓存同时服务盘中 provisional 和盘后 final 结果
- 前端高频轮询一个根本没重算的接口
- 每次查询都触发全量重算
- 收盘后仍继续展示旧的 provisional 结果为正式结论
- 不返回数据时间和缓存状态就输出交易建议
- 在规则版本变更后继续复用旧规则缓存

### 7.15 本规则的正式结论
系统默认采用“实时层短缓存、短周期层滚动刷新、日级层正式固化、周级层参数校准”的四层刷新模型。

也就是：
- 快数据快刷新
- 慢数据慢刷新
- provisional 与 final 分开缓存
- 事件变化优先局部重算
- 正式结果必须在盘后固化

任何没有时间戳、缓存状态、来源状态的结果，都不应被视为可靠的正式系统输出。

## 规则 8：API 字段规范、命名约定与状态语义规则

### 8.1 目标
系统必须统一 API 的字段命名、状态表达、时间字段、分页字段与错误语义，避免前后端各说各话、同一概念多种命名、同一状态多种口径，导致接口难维护、规则难复用、AI 难解释。

### 8.2 总体原则
系统默认遵循以下原则：

1. 同一业务概念只能有一套主字段名
2. 结构化字段优先，展示型文本字段后置
3. 枚举状态必须收敛，不允许同义词泛滥
4. 时间、状态、来源、质量字段必须显式给出
5. 面向前端、规则引擎、AI 层的字段语义必须一致

### 8.3 命名风格规则
API 字段默认统一使用 snake_case。

适用范围：
- JSON response 字段
- 请求参数字段
- 规则引擎输出字段
- 缓存对象字段
- 审计日志中的结构化字段

默认禁止混用：
- camelCase
- PascalCase
- 同一接口内 snake_case 与 camelCase 混用

例如：
- 正确：market_regime, signal_grade, data_cutoff_time
- 禁止：marketRegime, SignalGrade, dataCutoffTime

### 8.4 统一字段前缀与分组规则
系统推荐按语义分组命名：

1. market_*
   - 市场/指数层字段
2. sector_*
   - 板块/题材层字段
3. stock_*
   - 个股层字段
4. data_*
   - 数据时间、来源、质量、状态字段
5. signal_*
   - 信号等级、触发状态、确认状态字段
6. action_*
   - 执行动作、仓位、加减仓相关字段
7. ai_*
   - AI 摘要、AI 风险提示、一致性状态字段

这样可以保证：
- 接口可读性高
- 字段来源清晰
- 前端与 AI 层映射简单

### 8.5 时间字段规则
所有 API 中涉及时间的字段，默认必须统一语义。

推荐最小集合：
- data_cutoff_time
- generated_at
- updated_at
- next_refresh_after

规则要求：
1. data_cutoff_time 表示底层数据截至时间
2. generated_at 表示当前结果生成时间
3. updated_at 表示当前资源最后更新时间
4. next_refresh_after 表示前端建议下一次刷新时间

如果接口返回的是盘中结果，必须能看出：
- 数据到什么时候为止
- 结果是什么时候算出来的

### 8.6 状态字段规则
所有状态类字段必须采用有限枚举，不允许随意写自然语言状态。

推荐字段：
- data_status = provisional / final
- signal_type = intraday_signal / close_confirmed_signal
- confirmation_state = pending / confirmed / invalidated
- trading_status = trading / suspended / resumed
- cache_status = hit / miss / refreshed / stale
- execution_state = pending / active / reduced / exited

规则要求：
- 枚举值必须固定在文档中
- 新增枚举必须更新规则文档
- 前后端和 AI 层不得各自发明新说法

### 8.7 数值字段规则
所有分数、比例、仓位、涨跌幅字段必须显式说明量纲。

默认规则：
1. score 类字段统一为 0~100 或明确声明区间
2. ratio 类字段统一为 0~1，若返回百分比必须在字段名或文档中显式说明
3. pct / change_pct 类字段统一表示百分比变化值
4. position_cap_* 字段统一表示仓位上限比例

推荐示例：
- topic_heat_score = 0~100
- position_cap_single = 0~1
- market_breadth_ratio = 0~1
- change_pct = 百分比数值

### 8.8 文本字段规则
文本字段只用于解释，不得承担结构化判断责任。

推荐分层：
- title / label：短文本展示
- rule_based_advice：规则结论摘要
- ai_summary：AI 摘要解释
- risk_note / ai_risk_note：风险说明

默认禁止：
- 只返回一大段自然语言，不返回结构化字段
- 用文本描述替代 signal_grade / action_grade / invalidation

### 8.9 列表与分页规则
涉及列表接口时，必须统一分页和元数据结构。

推荐字段：
- items
- total
- page
- page_size
- has_more
- next_cursor（若采用游标分页）

默认规则：
1. 一个接口只能选择页码分页或游标分页中的一种主方式
2. items 中对象字段结构必须稳定
3. total 缺失时必须在文档中说明为什么缺失
4. 排序字段必须可解释

### 8.10 布尔字段规则
布尔字段必须采用 is_* / has_* / can_* / allow_* 风格，表达必须单一明确。

推荐示例：
- is_tradeable
- has_new_signal
- can_add_position
- allow_intraday_execution

默认禁止：
- flag1 / flag2 之类无语义命名
- maybe_trade 这类模糊命名

### 8.11 错误响应规则
所有 API 错误返回必须统一结构，不允许只返回裸字符串。

推荐最小结构：
- error_code
- error_message
- error_details
- request_id
- retryable

规则要求：
1. 业务错误和系统错误要区分
2. 可重试错误必须显式标记 retryable=true
3. 数据质量问题应尽量走结构化降级而不是直接 500
4. 规则冲突、数据缺失、状态不支持等应有明确错误码

### 8.12 版本与兼容规则
API 字段一旦对外使用，就必须考虑版本兼容。

默认规则：
1. 破坏性字段变更必须通过版本升级处理
2. 废弃字段应有过渡期
3. 同一概念的新旧字段不能长期并存
4. response 中若有 schema_version，应与规则版本可追踪关联

推荐字段：
- api_version
- schema_version
- rule_version

### 8.13 页面级字段最小集合
#### A. 页面一：市场总览
最小字段集合建议包含：
- market_regime
- market_risk_budget
- market_breadth_ratio
- data_cutoff_time
- data_status
- rule_based_advice
- ai_summary

#### B. 页面二：热点题材
最小字段集合建议包含：
- sector_name
- sector_strength
- sector_lifecycle
- leaders
- watchlist
- candidate_pool
- avoid_list
- data_cutoff_time

#### C. 页面三：候选/筛选
最小字段集合建议包含：
- stock_code
- stock_name
- signal_grade
- action_grade
- entry_trigger
- invalidation
- position_cap_single

#### D. 页面四：个股诊断
最小字段集合建议包含：
- stock_code
- stock_name
- stock_role
- signal_grade
- action_grade
- reduce_condition
- invalidation
- ai_summary

### 8.14 字段冲突与别名规则
默认不鼓励字段别名。

规则要求：
- 同一概念只保留一个主字段
- 若历史兼容必须保留别名，必须在文档中标记 deprecated
- 前端、AI 层、缓存层统一使用主字段名，不使用别名

### 8.15 禁止事项
以下做法在系统中默认禁止：
- 同一概念出现多个字段名
- 在接口里混用 snake_case 和 camelCase
- 返回没有时间戳和状态字段的交易结果
- 用自然语言替代结构化枚举
- 用前端临时拼装字段替代后端正式字段
- 错误响应只返回字符串而没有错误码
- API 改字段但不更新规则文档

### 8.16 本规则的正式结论
系统默认采用“统一 snake_case 命名、统一状态枚举、统一时间字段、统一错误结构、统一页面最小字段集合”的 API 规范。

也就是：
- 接口先讲结构
- 再讲展示
- 字段先统一
- 再做前端和 AI 适配

任何没有统一命名、状态语义和时间元数据的接口，都不应视为可长期维护的正式接口。

## 规则 9：日志、审计、回测与复盘口径规则

### 9.1 目标
系统必须保证所有正式交易结论都可回溯、可复盘、可审计、可用于回测验证，避免出现“当时为什么给出这个结论说不清”“历史结果无法复现”“实时与回测口径不一致”的问题。

### 9.2 总体原则
系统默认遵循以下原则：

1. 正式结论必须留痕
2. 留痕必须基于结构化字段，而不是只保存自然语言摘要
3. 回测口径必须与正式规则口径尽量一致
4. 复盘优先使用 final 数据与正式规则输出
5. 审计关注“当时输入了什么、规则怎么算的、输出了什么”三件事

### 9.3 日志分层规则
系统默认将日志分为四层：

1. 数据日志
   - 记录数据来源、数据时间、数据状态、补齐情况、质量状态

2. 规则日志
   - 记录规则引擎输入、关键中间结果、最终结构化输出

3. 执行日志
   - 记录动作等级、仓位上限、加减仓条件、失效条件、执行状态变化

4. 解释日志
   - 记录 AI 摘要、AI 风险提示、一致性状态

默认要求：
- 数据日志和规则日志是审计主依据
- 执行日志用于还原建议链路
- 解释日志只是辅助，不可替代前面三层

### 9.4 每条正式信号的最小审计字段
任何进入正式信号流的结果，至少必须保留以下字段：
- signal_id
- rule_version
- schema_version
- generated_at
- data_cutoff_time
- data_status
- data_sources
- market_regime
- sector_lifecycle
- stock_role
- signal_grade
- action_grade
- invalidation
- position_cap_single
- position_cap_portfolio
- ai_consistency_status

这样可以保证：
- 能追到是哪条信号
- 能追到用的是什么规则版本
- 能追到当时的数据状态和市场环境
- 能追到当时为什么是这个动作等级

### 9.5 事件日志与快照日志的区别
系统必须区分：

#### A. 事件日志
记录“发生了什么变化”。

适用内容：
- 信号由 pending -> triggered
- action_grade 从 observe -> probe
- 板块由 expansion -> climax
- 数据状态由 provisional -> final

#### B. 快照日志
记录“某个时点系统全貌是什么”。

适用内容：
- 某时刻市场总览快照
- 某时刻热点题材榜单
- 某时刻候选池与回避池
- 某时刻个股诊断结果

默认原则：
- 事件日志适合追过程
- 快照日志适合做复盘截面与回测切片

### 9.6 回测口径规则
回测默认只能使用：
- final 数据
- 正式规则字段
- 当时有效的规则版本

默认禁止：
- 用盘中 provisional 数据当作正式历史输入
- 用 AI 文本替代规则字段做回测判断
- 用当前新版本规则回算旧版本结果，却不记录版本差异

回测要求：
1. 明确数据口径（前复权/未复权）
2. 明确时间粒度（日线/周线/分钟线）
3. 明确规则版本
4. 明确样本范围与过滤条件

### 9.7 复盘口径规则
复盘默认使用：
- 当日 final 数据
- 当日正式规则输出
- 当日执行建议日志
- 当日关键盘中事件日志（若有保留）

复盘时应重点回答：
1. 当天市场环境是什么
2. 哪些板块被定义为主线/非主线
3. 哪些个股进入观察/候选/回避
4. 哪些信号盘中出现，哪些收盘确认
5. 哪些建议事后证明有效，哪些失效

### 9.8 审计链路规则
对于任一正式建议，系统都应能还原以下链路：

原始数据 -> 标准化数据 -> 规则引擎输入 -> 规则输出 -> 动作建议 -> AI 摘要

默认要求：
- 每一层都必须可追溯
- 关键层之间要有可关联 ID
- 不允许只保留最终页面文本，而无法追到规则字段

推荐关联字段：
- request_id
- signal_id
- snapshot_id
- trace_id

### 9.9 版本记录规则
日志、审计、回测、复盘都必须明确版本。

最小版本集合建议包含：
- api_version
- schema_version
- rule_version
- model_version（若涉及 AI 解释）
- data_pipeline_version

默认规则：
- 任何回测结果必须附带 rule_version
- 任何复盘报告必须附带 data_cutoff_time 与 rule_version
- 任何跨版本比较必须明确旧版本与新版本差异

### 9.10 统计与验收规则
系统必须能基于正式日志产出最基本的统计指标。

最小统计集合建议包括：
- 各信号等级数量分布
- 各动作等级数量分布
- 候选池命中率
- 回避池规避率
- 不同市场环境下的信号表现
- 不同板块生命周期下的信号表现
- 盘中信号与收盘确认信号的一致率

这些统计只能建立在正式结构化日志之上。

### 9.11 日志保留规则
不同日志层可以采用不同保留期，但正式审计字段保留期应最长。

默认建议：
- 实时事件日志：可较短保留
- 盘中过程日志：中期保留
- 日级正式结果日志：长期保留
- 回测结果与复盘报告：长期保留

默认原则：
- 越接近正式结论，保留越久
- 越接近临时过程，可适当压缩

### 9.12 AI 日志规则
AI 相关日志只能作为辅助材料。

默认要求：
- 保留 ai_summary
- 保留 ai_risk_note
- 保留 ai_consistency_status
- 保留 model_version（若可得）

默认禁止：
- 只存 AI 文本，不存规则字段
- 用 AI 文本反推正式交易建议
- 审计时以 AI 摘要替代规则证据

### 9.13 错误与异常审计规则
系统中的异常也必须可审计，而不是只在控制台一闪而过。

最小异常字段建议包含：
- error_code
- error_message
- request_id
- trace_id
- affected_scope
- retryable
- occurred_at

适用场景：
- 数据源失败
- 规则字段缺失
- 状态冲突
- 缓存异常
- AI 与规则冲突

### 9.14 禁止事项
以下做法在系统中默认禁止：
- 只有页面展示结果，没有结构化审计日志
- 只保存最终 AI 摘要，不保存规则字段
- 回测使用与正式系统不同的规则口径却不说明
- 复盘时混用 provisional 与 final 结果
- 无法根据日志还原当时的输入、规则版本和输出
- 统计建立在自然语言文本而不是结构化字段上

### 9.15 本规则的正式结论
系统默认采用“正式结果结构化留痕、回测只认 final 与规则字段、复盘以正式输出为准、审计可还原完整链路”的日志与审计体系。

也就是：
- 正式信号必须可追溯
- 回测必须可复现
- 复盘必须口径一致
- 审计必须能解释当时为什么得出这个结论

任何不能追溯到数据、规则版本和结构化输出的结论，都不应视为可信的正式系统结论。

## 规则 10：本地通达信数据接入与系统存储边界规则

### 10.1 目标
系统必须明确“本地通达信原始数据”和“系统内部持久化数据”之间的边界，避免出现整库复制、双份原始数据漂移、同步复杂度失控，或后续无法判断哪一份数据才是正式依据的问题。

### 10.2 总体原则
系统默认遵循以下原则：

1. 本地通达信目录是原始数据源之一，不是系统私有数据库
2. 系统默认对本地通达信数据采用只读直连方式使用
3. 系统不复制整份通达信原始目录作为主数据仓库
4. 系统只保存标准化结果、派生结果、正式快照和审计结果
5. 若未来有独立部署、长期归档或高性能回测需求，再引入增量导入机制

### 10.3 默认接入方式
系统默认采用“只读接入 + 标准化缓存 + 正式归档”的三层接入方式。

#### A. 原始层
- 直接读取本地通达信目录
- 目录默认是：`/mnt/c/new_tdx64`
- 原始文件保持不改写
- 原始层只作为源，不作为系统业务结果输出层

#### B. 标准化层
- 将本地通达信读取结果转成统一 schema
- 增加统一元数据字段
- 允许短期缓存
- 供规则引擎和 API 层直接消费

#### C. 正式归档层
- 仅保存系统真正需要的正式结果
- 不保存整库原始镜像
- 用于回测、复盘、审计、长期留存

### 10.4 适合直接读取的本地数据类型
以下数据默认适合直接从本地通达信目录读取：
- 日线 `.day`
- 1 分钟线 `.lc1`
- 板块/自定义板块文件
- 本地已有财务/缓存辅助文件（如适用且字段可解释）

规则要求：
- 直接读取时必须标记 data_source=local_tdx
- 直接读取结果进入系统后必须先经过标准化层
- 不允许原始文件路径直接暴露给前端业务层作为正式接口契约

### 10.5 不建议直接依赖的本地数据类型
以下数据默认不建议作为长期稳定主源直接依赖：
- 覆盖不稳定或内容偏旧的本地 5 分钟文件
- 来源不稳定、口径不透明的临时缓存文件
- 无法稳定解释字段含义的通达信内部临时文件

默认处理方式：
- 可读取做辅助检查
- 但不直接作为主规则引擎正式输入
- 如业务需要，应由更可靠的底层数据重新聚合生成

### 10.6 系统内部应保存什么
系统内部默认只保存以下几类数据：

1. 标准化缓存结果
   - 对原始通达信数据做统一字段映射后的临时结果

2. 聚合派生数据
   - 例如由 1 分钟聚合得到的 5/15/30/60 分钟结果
   - 或统一复权口径后的可复用派生序列

3. 正式快照
   - 日级 final 快照
   - 市场环境快照
   - 板块热度快照
   - 个股诊断快照

4. 规则输出与执行结果
   - 候选池/观察池/回避池
   - signal_grade / action_grade
   - 加减仓条件、失效条件

5. 审计与回测所需结构化日志
   - 供回测、复盘、审计长期使用

### 10.7 系统内部不应保存什么
系统默认不应保存：
- 通达信原始目录的整库复制副本
- 与通达信原始文件一一镜像的重复原始文件集合
- 没有标准化和业务价值的临时原始镜像

原因：
- 双份原始数据易漂移
- 难以判断哪份才是准的
- 同步成本高
- 存储浪费
- 审计链路反而变模糊

### 10.8 派生周期数据规则
如果系统需要稳定使用 5/15/30/60 分钟周期，默认不直接依赖通达信现成本地高周期分钟文件，而应优先采用：
- 1 分钟原始线 -> 系统内聚合 -> 派生周期持久化

这样做的好处：
- 周期生成规则可控
- 口径统一
- 覆盖稳定
- 可追溯聚合逻辑

### 10.8A 5 分钟线作为主派生基底的补充条款
当本地 5 分钟线经过验证满足以下条件时，可将其提升为 15/30/60 分钟周期的默认主派生基底：
- 最新交易日与可靠的 1 分钟线最新交易日一致
- 覆盖长度稳定
- 大部分目标标的读取正常
- 时间戳连续、无明显缺洞
- 抽样验证通过率达到可接受水平

在当前用户环境 `/mnt/c/new_tdx64` 的最新实测结论中：
- 沪深主市场样本的 5 分钟线已经追平到最近交易日 `2026-04-24`
- 抽样可正常读取的沪深标的中，5 分钟线与 1 分钟线最新交易日匹配率达到 `100%`
- 北交所及部分特殊代码仍存在读取为空或不稳定情况

因此当前默认口径应定为：
1. 沪深主市场：
   - 5 分钟线可作为 15/30/60 分钟的默认主派生基底
2. 北交所及特殊代码：
   - 暂不纳入“已验证可靠的 5 分钟主派生基底”范围
3. 全市场统一规则：
   - 必须始终保留 1 分钟线作为底层回退源

推荐派生链路：
- 5 分钟 -> 15 分钟
- 5 分钟 -> 30 分钟
- 5 分钟 -> 60 分钟

默认禁止：
- 在 5 分钟线未验证覆盖的标的范围内强行统一使用 5 分钟作为唯一派生基底
- 去掉 1 分钟回退机制
- 不区分沪深主市场与北交所/特殊代码就宣称“全市场 5 分钟主派生基底已完全可用”

### 10.9 独立部署与长期归档例外规则
只有在以下场景下，系统才建议引入“增量导入”机制：

1. 未来部署到没有本地通达信目录的服务器
2. 需要多年级别的历史冻结归档
3. 需要高性能批量回测
4. 需要与通达信目录物理隔离的正式备份

即便如此，也应采用：
- 增量导入
- 标准化落库
- 可追踪版本

而不是：
- 机械复制整个 `vipdoc/` 目录作为主仓库

### 10.10 数据一致性规则
系统必须始终能回答“当前这条结果来自哪一层”。

推荐字段：
- data_source = local_tdx / tdx_api / public_api / derived_store
- storage_layer = raw / standardized_cache / derived_store / final_archive
- source_file_hint
- derived_from

默认要求：
- 原始直读结果要标明 source_file_hint（如适用）
- 派生结果要标明 derived_from
- 正式归档结果要标明是否来自 local_tdx 直读链路或派生链路

### 10.11 性能与可靠性平衡规则
系统读取本地通达信时，默认优先保证：
1. 正确性
2. 可追溯性
3. 一致性
4. 性能

因此：
- 高频查询可走标准化缓存
- 高频规则可走派生持久化结果
- 但原始口径定义仍以只读本地源为准

### 10.12 迁移与替换规则
未来如果需要把部分本地通达信数据迁入系统数据库，必须满足：
- 有明确业务目的
- 有明确字段映射规则
- 有版本控制
- 有回滚方案
- 有源数据与导入数据一致性校验

默认禁止无目的地“先全部复制再说”。

### 10.13 禁止事项
以下做法在系统中默认禁止：
- 把整个通达信目录复制为系统主数据仓库
- 修改本地通达信原始文件
- 不区分原始层、标准化层、归档层而混合使用
- 用系统复制副本覆盖本地通达信真实源数据
- 不记录来源层级就输出正式结果
- 为了图方便把所有原始文件全部导入数据库但不做字段治理

### 10.14 本规则的正式结论
系统默认采用“本地通达信原始数据只读直连、系统仅保存标准化缓存/派生数据/正式快照/审计结果”的存储边界模型。

也就是：
- 原始数据直接读
- 不整库复制
- 系统只存真正有业务价值的结果
- 若未来需要独立部署、长期归档或高性能回测，再增加增量导入机制

任何把通达信原始目录机械复制成第二套主数据仓库的方案，都不应视为默认推荐方案。
