# 代码目录与运行数据目录边界说明

## 目标
明确哪些内容属于“应该进 Git 的代码/文档资产”，哪些属于“只在本地或数据存储层存在的运行数据资产”，避免：
- 聚合数据误提交进仓库
- 归档结果污染代码仓库
- 代码与数据边界模糊

## 总体原则
1. 代码、文档、配置进 Git。
2. 真实运行数据、归档数据、审计数据不进 Git。
3. 示例文件可以进 Git，但真实日常产出不进 Git。
4. 原始通达信数据不属于本仓库的一部分。

## 应进入 Git 的内容
以下内容默认应进入 Git：

### 1. 代码
- `app/`
- `scripts/`
- 后续的 `api/` / `web/` / 工具脚本

### 2. 文档
- `docs/*.md`
- 设计文档
- 规则文档
- 归档方案文档
- 指标草案文档

### 3. 示例文件
- `docs/*.example.json`
- 用于说明结构的示例 JSON
- 小型 schema 示例

### 4. 配置与清单
- `.gitignore`
- README
- 后续配置模板

## 不应进入 Git 的内容
以下内容默认不应进入 Git：

### 1. 运行时数据
- `data/standardized/`
- `data/derived/`
- `data/archive/`
- `data/audit/`
- 所有按交易日生成的真实归档结果

### 2. 临时产物
- lock 文件
- dry-run 产物
- 本地测试生成数据
- 缓存文件
- 中间计算结果

### 3. 本地原始数据源
- `/mnt/c/new_tdx64`
- 通达信原始目录
- 任何本地镜像副本

## 推荐目录边界

### 仓库内：放什么
```text
Project-Hermes-Stock/
├─ app/
├─ scripts/
├─ docs/
├─ references/
└─ .gitignore
```

### 仓库内但不进 Git：放什么
```text
Project-Hermes-Stock/
└─ data/
   ├─ standardized/
   ├─ derived/
   ├─ archive/
   └─ audit/
```

## 当前推荐 .gitignore 策略
当前建议直接忽略：
```text
/data/
```

含义：
- `data/` 下真实运行数据都不进 Git
- 文档示例留在 `docs/` 下，由 Git 管理

## 为什么示例文件放 docs/ 而不是 data/
原因：
1. 示例文件属于“说明资料”，不是运行数据
2. 放在 `docs/` 下更清晰
3. 可以稳定进入 Git
4. 不会和真实归档目录混淆

## 对你当前项目的推荐做法
### Git 管理
用于同步：
- 规则
- 文档
- 脚本
- 代码骨架
- 示例 JSON

### data 目录
用于本机运行：
- 聚合 bars
- 派生 features
- datasets
- archive
- audit

### 外部原始源
用于只读接入：
- 本地通达信原始数据目录

## 结论
当前项目推荐采用三层边界：

1. 仓库资产层（进 Git）
- 代码
- 文档
- 配置
- 示例

2. 运行数据层（不进 Git）
- `data/` 下所有真实聚合、归档、审计结果

3. 外部原始源层（不进 Git，也不纳入仓库）
- 通达信本地原始目录

这样后续你执行 Git 同步时，代码和设计会同步，但真实运行产生的数据不会跟着仓库一起走。
